# 프로젝트 목적

READi ERP는 단순한 ERP 시스템이 아니다.

회사의 업무를 표준화하고,
데이터를 하나로 연결하며,
운영체계를 구축하기 위한 프로젝트이다.

ERP는 목적이 아니라 수단이다.

모든 개발은

업무 표준화
→ 데이터 일원화
→ 업무 자동화
→ 의사결정 지원

을 목표로 한다.

---

# 프로젝트 철학

작게 시작한다.

쉽게 이해한다.

쉽게 유지보수한다.

필요한 만큼만 만든다.

사람보다 시스템이 기억한다.

데이터는 한 번만 입력한다.

프로세스를 먼저 설계하고
시스템은 마지막에 만든다.

# 개발 전 반드시 생각할 것

기능을 만들기 전에 항상 아래 순서대로 검토한다.

1. 목적

왜 필요한가

2. 현재 방식 (AS-IS)

현재 업무는 어떻게 진행되는가

3. 표준 방식

일반 제조기업에서는 어떻게 운영하는가

4. READi 방식 (TO-BE)

30명 규모 스타트업에 맞게
최소한의 절차로 어떻게 운영할 것인가

5. 역할

누가 입력하는가

누가 승인하는가

누가 사용하는가

6. 데이터

어떤 데이터가 필요한가

중복 입력은 없는가

다른 기능에서도 사용할 수 있는가

7. 통제

승인

권한

변경이력

상태관리

누락방지

8. 시스템

마지막으로 시스템을 구현한다.

UI는 가장 마지막이다.

---

# 스타트업 운영 원칙

READi는 약 30명 규모의 스타트업이다.

대기업 ERP를 그대로 따라하지 않는다.

복잡한 승인 절차

불필요한 입력

과도한 권한관리

사용하지 않는 기능

은 만들지 않는다.

현재 필요한 만큼만 개발한다.

회사 성장에 맞춰 확장 가능한 구조를 만든다.

---

# 데이터 원칙

Single Source of Truth

동일한 정보는 한 번만 입력한다.

Master Data와
Transaction Data를 분리한다.

Airtable에 종속되지 않는다.

언제든 SQL로 이전 가능해야 한다.

모든 데이터는

검색

통계

자동화

확장이 가능해야 한다.

기능부터 만들지 않는다.

프로세스를 먼저 만든다.

목적

담당자

출력

상태

변경이력

을 가진다.

# 내부통제 원칙

승인

권한

감사 추적

단, 회사 규모에 맞지 않는 과도한 통제는 만들지 않는다.

---

# 개발 원칙

기존 구조를 우선 활용한다.

Business Logic은 Page에 작성하지 않는다.

Repository 패턴을 따른다.

중복 코드를 만들지 않는다.

공통 기능은 재사용한다.

사용하지 않는 기능은 개발하지 않는다.

읽기 쉬운 코드를 작성한다.

---

# UI 원칙

UI보다 업무가 우선이다.

입력은 최소화한다.

검색을 우선한다.

자동화를 우선한다.

사용자가 적게 클릭할수록 좋은 시스템이다.
